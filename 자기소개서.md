Q1. 개인적 성장 경험
"지금까지 살면서 가장 큰 변화나 성장을 경험한 순간이 있다면 자유롭게 이야기해주세요. 어떤 계기로 그런 변화가 시작되었고, 그 과정에서 어떤 생각과 감정을 겪었는지, 그리고 그 경험이 지금의 당신에게 어떤 영향을 미쳤는지 편하게 작성해주세요."

개발자로의 전환은 제 인생에서 가장 큰 성장이었고, 그 시작은 우연이었습니다.

뚜렷한 목표 없이 시간을 보내던 중 연히 무료 국비 코딩 교육 프로그램을 접했고, "공짜니까 들어나 보자"라는 가벼운 마음으로 참여했습니다.

그런데 막상 코딩을 배우며 처음으로 진짜 열정을 발견했습니다. 내 손으로 무언가를 만드는 재미, 누군가 그것을 사용하는 기쁨, 문제를 효율적으로 해결했을 때의 성취감. 밤새 공부해도 질리지 않는 경험이었습니다. 단순한 흥미를 넘어, 문제를 설계하고 해결하는 과정 자체가 즐거운 일이라는 사실을 처음 실감했습니다.

하지만 늦은 나이에 시작하는 것이 맞는지 확신이 필요했습니다. "이 열정이 일시적인 감정이 아닌가?" 스스로를 검증하기 위해 이전 직장에 복귀했습니다. 매일 퇴근 후부터 새벽까지 개발 공부를 하는 루틴을 1년간 지속했습니다. 동시에 GitHub 365일 커밋이라는 구체적 목표를 세웠습니다.

1년 후, 목표를 달성했을 때 비로소 확신했습니다. 이것이 평생 열정을 쏟을 수 있는 일이라고 말입니다. 그 시점에 개발자 전향을 결정했습니다.

3년이 지난 지금도 열정은 여전합니다. 2025년 두 번째 365일 커밋 달성을 한 달 앞두고 있습니다. 하루 15시간 근무하는 날에도 어김없이 개발 공부와 블로그 작성, 사이드 프로젝트를 병행합니다.

개발자로의 전환은 단순한 직업 변경이 아니라, 평생 몰입할 수 있는 일을 찾은 가장 큰 성장이었습니다. 지금도 매일 새로운 기술을 학습하고, 문제를 해결하는 과정에서 같은 열정을 느낍니다.

Q2. 인간관계 경험
"다른 사람들과 함께 일하거나 생활하면서 기억에 남는 경험이 있다면 이야기해주세요. 좋았던 경험이든 어려웠던 경험이든 상관없습니다. 그 상황에서 어떤 일이 있었고, 당신은 어떻게 대응했으며, 그 경험을 통해 무엇을 배웠는지 솔직하게 작성해주세요."

10여개 팀과의 협업에서 각 전문가의 시각을 배우는 것이 얼마나 중요한지 깨달았습니다.

대기업 제약회사 AI 프로젝트에서 AI 파트 리더로 약 10개 팀의 전문가들과 협업했습니다. 컨설턴트, 인프라 전문가, RAG 개발팀, 백엔드/프론트 개발팀, QA, 보안팀, DBA, 부하 테스트 전문가 등 각 분야 최고 전문가들과 함께 일했고, 3개월마다 교체되는 15명의 인턴급 개발자도 관리했습니다.

BCG 컨설턴트에게서는 문제 분해 사고방식을 배웠습니다. "MR 생산성 향상"이라는 큰 문제를 4R 전략(Right Target, Right Product, Right Message, Right Timing)으로 세분화하는 방식이었습니다. 처음에는 불필요하다고 여겼으나, 복잡한 AI 시스템 설계 시 매우 효과적임을 깨달았습니다.

인프라 전문가에게서는 프로세스의 중요성을 배웠습니다. 방화벽 하나 개방에 5일이 소요된다는 사실을 처음엔 이해하기 어려웠습니다. 그러나 대기업에서는 보안, 네트워크, SG 설정 담당자가 분리되어 있고 각각 승인 프로세스가 존재했습니다. 사전 담당자 파악과 조율이 얼마나 중요한지 체득했습니다.

QA 전문가는 재현 가능성의 중요성을 알려주었습니다. Jira 티켓마다 정확한 입력값, 재현 방법, 환경 정보를 요청했습니다. 이 습관은 이후 200개 이상의 테스트 케이스 작성 시 큰 도움이 되었습니다.

어려움도 있었습니다. 10개 팀과의 협업으로 하루 3~4개 회의가 일상이었고, 대기업 특유의 "변경 회피 문화"로 개선 시도마다 부담이 컸습니다. 게다가 인턴 개발자 15명 관리와 한국자동차연구원 프로젝트를 동시 진행하며 물리적 시간 부족에 직면했습니다.

해결책은 "시스템을 통한 해결"이었습니다. 인력 교체가 불가피하므로 아키텍처를 재설계하여 Agent 구조, init 함수, tool 정의까지 완성된 템플릿을 제공했습니다. Jira 티켓에는 재현 방법을 상세히 기록하여 누구나 이해할 수 있도록 했습니다.

이 경험을 통해 깨달은 것은 "전문가의 시각을 배우는 것"의 가치였습니다. 당연하다고 생각한 것이 다른 팀에게는 큰 리스크가 될 수 있었습니다. "좋은 아키텍처"와 "안전한 구조"는 다를 수 있었습니다. 각 전문가의 관점에서 문제를 바라보니 더 나은 해결책이 도출되었습니다.

무언가를 설계할 때 "내 방식이 최선인가"를 항상 성찰하게 되었습니다. 다양한 시각에서 문제를 바라보는 습관이 형성되었습니다.

Q3. 의미 있는 프로젝트나 활동
"개인적으로 의미가 있었거나 보람을 느꼈던 프로젝트, 활동, 또는 경험이 있다면 자유롭게 소개해주세요. 왜 그 일을 시작하게 되었고, 어떤 과정을 거쳤으며, 어떤 결과나 변화가 있었는지 편하게 작성해주세요."



LangChain Open Tutorial 오픈소스 프로젝트 기여가 가장 의미 있었습니다. 2,000개 이상의 커밋이 발생한 대규모 오픈소스로, 한국어 개발자들이 RAG와 LangGraph를 배울 수 있도록 돕는 튜토리얼 플랫폼입니다.

대기업 제약회사와 한국자동차연구원 프로젝트에서 RAG와 Multi-Agent 시스템을 구축하며 많은 것을 학습했습니다. 그러나 한국어 참고 자료가 매우 부족했습니다. 영문 문서를 읽으며 시행착오를 겪고, 에러를 해결하고, 다시 시도하는 과정을 반복했습니다. 그 과정에서 생각했습니다. "학습한 내용을 다른 개발자들과 공유한다면 어떨까?"

단순한 선행이 아니었습니다. 업무에서 습득한 지식을 사회에 환원하고 싶었습니다. 누군가 겪은 시행착오를 거치지 않고 더 빠르게 학습할 수 있다면, 그것이 진정으로 의미 있는 일이라고 판단했습니다.

약 7주간 3개의 튜토리얼을 작성했습니다.

1. LlamaParse 튜토리얼: PDF, Word, Excel 등 다양한 파일을 파싱하는 법
2. Conversation Memory Management: LangGraph 기반 챗봇 메모리 관리 시스템
3. CoT-Based Smart Web Search: Chain-of-Thought 기반 스마트 웹 검색

각 튜토리얼은 단순히 코드만 제공하는 것이 아니라, "설계 의도"와 "해결하려는 문제"를 명확히 설명했습니다. 실무에서 겪은 문제들(토큰 제한 초과, 메모리 관리 이슈 등)을 예시로 제시하여 독자들이 실전에서 즉시 활용할 수 있도록 구성했습니다.

작성 과정은 예상보다 어려웠습니다.

영문 튜토리얼을 한국어로 번역하는 것은 단순 번역이 아니었습니다. "이 개념을 한국 개발자들이 어떻게 이해할 것인가"를 끊임없이 고민했습니다. 전문 용어의 사용, 의역, 예시 추가 등을 매번 신중하게 판단해야 했습니다.

또한 Peer Review 과정에서 다른 기여자들의 피드백을 통해 설명의 명확성과 부족한 부분을 파악했습니다. "이 부분은 초보자가 이해하기 어려울 것"이라는 피드백을 받으면 예시를 추가하거나 설명을 더 상세히 작성했습니다.

결과는 매우 보람 있었습니다.

GitBook과 GitHub에 게시된 튜토리얼에 대해 "RAG 개념 이해에 큰 도움이 되었습니다", "실전에서 즉시 적용했습니다"와 같은 피드백을 받았습니다.

가장 큰 보람은 "기술 지식의 민주화"에 기여했다는 점입니다. 튜토리얼을 통해 누군가는 더 빠르게 학습하고, 더 나은 서비스를 개발할 것입니다. 그들이 또 다른 누군가를 도울 것입니다. 이러한 선순환이 지속되면 한국 개발자 생태계가 더욱 성장할 것이라 믿습니다.

이 경험은 개발 철학에도 영향을 미쳤습니다. 코드 작성 시 "6개월 후 이 코드를 보는 사람이 이해할 수 있는가"를 고민하게 되었습니다. 주석에는 설계 의도를 명확히 기록하려 노력합니다. 기술 블로그(https://syshin0116.github.io)에도 프로젝트 회고와 문제 해결 과정을 꾸준히 기록합니다.

Q4. 문제 해결 경험
"일상생활이나 학업, 업무에서 복잡하거나 애매한 문제를 만난 적이 있나요? 그 문제를 어떻게 파악하고 해결해나갔는지, 그 과정에서 어떤 시행착오가 있었는지 자신의 경험을 자유롭게 이야기해주세요. 큰 문제가 아니어도 괜찮습니다."



GPT가 제품명을 오표기하는 문제가 발생했습니다. "토스젯에이"를 "토스젯A"로 바꾸거나, "토스싹에이", 심지어 "토스민토스" 같은 존재하지 않는 제품명으로 생성하기도 했습니다. 제약 도메인에서 제품명 오류는 치명적입니다. 고객사 만족도 99%였으나, 나머지 1%의 불만족이 전부 이 문제였습니다.

문제는 애매했습니다. 전체 답변 중 극소수(약 1~2%)에만 발생하는 현상이었습니다. 팀 내에서도 "이 정도는 감수해도 되지 않나"는 의견이 많았습니다. 하지만 제약 도메인에서 제품명 하나의 오류가 의료 사고로 이어질 수 있다고 판단했습니다.

프롬프트 개선을 시도했지만 효과가 없었습니다. "왜 특정 단어만 오표기되는가" 팀원들과 브레인스토밍을 시작했습니다.

먼저 토크나이저 레벨까지 분석했습니다. "토스젯에이"가 (토스)(젯)(에이) 3개 토큰으로 분할되고, GPT는 학습 데이터에서 더 빈번한 "A"를 예측한다는 것을 발견했습니다.

문제를 정리해 OpenAI 개발자와 LLM 전문가들을 인터뷰했습니다. 여러 해결 방안이 나왔습니다:
1. 커스텀 토크나이저 도입 (제품명을 하나의 토큰으로)
2. 파인튜닝 (제약 도메인 데이터로 재학습)
3. 후처리 로직 (제품명 사전 기반 자동 교정)
4. 컨텍스트 강화 (프롬프트에 제품명 리스트 추가)

각 방안을 평가했습니다. 커스텀 토크나이저와 파인튜닝은 근본적 해결책이지만 출시 일정(2주)에 불가능했습니다. 현실적 제약을 고려해 3번과 4번을 조합하기로 결정했습니다.

테스트를 진행했습니다:
- 후처리만: 오류율 60% → 15%
- 컨텍스트 강화만: 오류율 60% → 25%
- 두 방식 조합: 오류율 60% → 8%
- 검증 단계 추가: 오류율 8% → 3%

결과적으로 조합 방식이 최적임을 확인하고 적용했습니다.

이 경험에서 배운 것은 집단 지성의 힘입니다. 혼자였다면 프롬프트 개선만 시도했을 것입니다. 팀 브레인스토밍으로 토크나이저 레벨 분석이라는 방향을 잡았고, 전문가 인터뷰로 다양한 해결책을 얻었으며, 체계적 테스트로 상황에 맞는 최적 솔루션을 찾을 수 있었습니다.

Q5. 직접 만들거나 실행해본 경험
"무언가를 직접 만들어보거나 아이디어를 실행에 옮겨본 경험이 있다면 이야기해주세요. 간단한 것부터 복잡한 것까지 어떤 것이든 괜찮습니다. 왜 그것을 만들거나 실행하고 싶었는지, 어떤 과정을 거쳤는지, 결과는 어땠는지 자유롭게 작성해주세요."

작은 불편함을 해결하기 위해 만든 챗봇이 주변 사람들의 생산성까지 향상시켰습니다.

두 가지 불편함이 있었습니다. 첫째, 여기저기서 받은 유용한 링크들을 체계적으로 관리하고 싶었습니다. 카카오톡으로 받은 기술 블로그 링크를 "나중에 읽어야지" 하다가 잊어버리고, 북마크에 저장해도 결국 같은 내용을 다시 검색했습니다. 둘째, 카톡에서도 GPT를 쓰고 싶었습니다. 시험 단톡방에서 사람들의 질문을 받아줄 LLM이 있으면 좋겠다고 생각했습니다.

PodlyBot을 만들었습니다. 안드로이드 공기계에 메신저봇R을 설치했습니다. 작동 방식은 간단합니다:
1. 카톡방에 URL 게시
2. 봇이 자동 감지
3. LLM으로 내용 요약
4. Notion에 자동 저장

예상치 못한 반응이 있었습니다. 지인들이 자신의 카톡방에도 초대해달라고 요청했습니다. URL 정리뿐 아니라 LLM 대화 기능도 별도 앱 설치 없이 카톡에서 바로 쓸 수 있어 편리하다고 했습니다.

회사에서도 LabQ Bot을 만들었습니다. 회사 카톡방의 기술 자료를 자동으로 정리하고, 고객사가 환경 설정 없이 카톡으로 우리 AI 서비스를 테스트할 수 있게 했습니다. "이 기능 어때요?" 하면 즉시 시연 가능합니다. 안드로이드 공기계로 24/7 운영 중입니다.

기술적으로 까다로웠던 부분은 Rhino JavaScript 엔진의 한계였습니다. 최신 문법을 사용할 수 없어 async/await 대신 콜백 패턴으로 구현했고, 카톡 알림 기반 이벤트 처리와 채팅방별 컨텍스트 관리를 해결해야 했습니다.

결과는 만족스러웠습니다. 개인 불편함 해결이 목적이었지만, 지인들과 회사 팀원들의 생산성까지 향상시켰습니다. 수익 창출이 아니라 실제 문제 해결에 집중했기에 가능했습니다.

Q6. 효율성 개선 경험
"반복적이거나 비효율적이라고 생각되는 일을 더 나은 방법으로 바꿔본 경험이 있나요? 일상생활, 학습, 업무 등 어떤 영역이든 상관없습니다. 어떤 문제를 발견했고, 어떻게 개선했으며, 그 결과는 어땠는지 편하게 이야기해주세요."

대기업 제약회사 프로젝트 초기, 챗봇 응답 시간이 평균 3분 이상 걸렸고 1,000개 질문 중 30%만 답변 가능했습니다. 전체 파이프라인을 최적화하여 이 문제를 해결했습니다.

먼저 프로파일링 툴로 병목 지점을 측정했습니다. Vector DB 검색이 1분 30초로 가장 느렸고, LLM 호출이 40초, MariaDB 쿼리가 30초였습니다.

단계별로 접근했습니다. 첫 번째로 LanceDB를 Qdrant로 교체했습니다. 하이브리드 검색을 지원하여 검색 시간이 1분 30초에서 20초로 단축되었습니다. 두 번째로 Redis 캐싱을 도입했습니다. "이번 달 매출" 같은 반복 질문을 1시간 TTL로 캐싱하여 즉시 응답이 가능해졌습니다. 세 번째로 복잡한 질문에서 MariaDB, Qdrant, Redshift를 동시 조회하는 병렬 처리를 적용하여 3분을 50초로 줄였습니다. 마지막으로 LangGraph Multi-Agent Supervisor로 워크플로우를 최적화했습니다. 간단한 질문은 Chain으로 5~10초에, 복잡한 질문은 Multi-Agent로 30~50초에 처리하도록 했습니다.

결과는 만족스러웠습니다. 응답 속도를 83% 개선하여 3분에서 30초로 단축했고, 답변 커버리지는 233% 향상되어 30%에서 100%가 되었습니다. 고객 만족도 99%를 달성했습니다.

이 경험에서 배운 핵심은 데이터 기반 의사결정입니다. 추측이 아니라 프로파일링 결과로 병목을 파악하고 우선순위를 정했습니다. 그리고 한 부분만이 아니라 전체 파이프라인을 최적화해야 진정한 성능 개선이 가능하다는 것을 깨달았습니다.

Q7. 프로젝트나 목표 달성 경험
"하나의 프로젝트나 목표를 처음부터 끝까지 완주해본 경험이 있다면 이야기해주세요. 계획을 세우는 것부터 실행하고 마무리하는 과정까지, 어떤 어려움이 있었고 어떻게 극복했는지, 최종적으로 어떤 성과나 배움이 있었는지 자유롭게 작성해주세요."

한국자동차연구원 프로젝트는 2024년 3월부터 2025년 11월까지 1년 8개월간 진행한 가장 긴 프로젝트입니다.

1차 3개월은 MVP로 기본 RAG 챗봇을 구축했습니다. 2차 4개월은 파일 관리와 토큰 추적 기능을 확장했습니다. 3차 6개월은 대규모 재설계로 Flask를 FastAPI로 전환하고 LangGraph 멀티 에이전트를 구축했습니다. 4차 5개월은 멀티 파일 파서 통합과 시각화를 고도화했습니다.

가장 큰 어려움은 3차에서 백엔드 전체를 재작성하는 것이었습니다. 7개월 동안 구축한 시스템을 폐기하는 것이 부담스러웠지만, 일괄 전환은 위험하다고 판단했습니다. API 문서를 먼저 작성하고 프론트엔드팀과 합의한 뒤, 인증부터 파일 관리, 챗봇, 보고서 생성까지 모듈별로 하나씩 이전하며 각각 테스트했습니다. 큰 사고 없이 마이그레이션을 완료했습니다.

3차부터 3명 팀의 PL이 된 것도 새로운 도전이었습니다. 팀원들이 제 코드를 이해하지 못해 설명에 시간이 더 걸렸습니다. 문제는 제 코드에 있었습니다. API 설계 문서를 상세히 작성하고, 200개 이상의 테스트 케이스로 시스템 안정성을 확보하며, 코드 리뷰 시간을 확보해 팀원 성장을 지원하는 방식으로 접근을 바꿨습니다.

또한 한국자동차연구원과 제약회사 프로젝트가 중복되어 매일 새벽 2시까지 근무하며 한계에 도달했습니다. 우선순위를 명확히 분리했습니다. 오전은 제약회사 회의와 긴급 이슈, 오후는 자동차연구원 개발, 저녁은 제약회사 코드 리뷰와 아키텍처 설계로 시간을 나눴습니다.

결과는 만족스러웠습니다. 보고서 작성 시간을 1시간에서 1분으로 98% 단축했고, 1인 개발에서 3명 팀 PL로 성장했습니다. Flask를 FastAPI로, ChromaDB를 Qdrant로 재설계 완료했으며, LangGraph 멀티 에이전트 아키텍처를 구축하고 4개 파일 파서를 통합했습니다.

이 프로젝트에서 배운 것은 처음부터 완벽할 수 없다는 것입니다. 1차 때 Flask는 최선의 선택이었지만 3차에서 FastAPI로 전환해야 했습니다. 현재 가능한 최선을 실행하고 지속적으로 개선하면 됩니다. 1년 8개월 동안 보고서 작성 시간이 1시간에서 30분, 10분, 1분으로 단계별 개선되며 98% 단축이라는 큰 성과가 되었습니다. 개인의 빠른 성과보다 팀의 지속 가능한 성장이 더 가치 있다는 것도 깨달았습니다.

델타아이에서 온톨로지 엔진 개발도 긴 여정일 것입니다. 단계별로 개선하며 완주하면 "전문가의 20년 경험을 누구나 활용할 수 있는" AGI를 만들 수 있을 것이라 믿습니다. 그 여정을 델타아이 팀과 함께하고 싶습니다.
